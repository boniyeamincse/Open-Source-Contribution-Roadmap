# Module 3 Examples: Anatomy of a Repository

Let's look at how a real-world repository (like this book!) is structured.

## 📖 The "Good First Issue" Label
Many projects use labels to help newcomers. Here's what they look like in the code (meta-data):
- **Name**: `good first issue`
- **Color**: Often green or light blue.
- **Description**: "Newcomer-friendly issues that require minimal context of the entire codebase."

## 📜 Choosing a License
Different projects have different needs.
- **Example A**: A developer wants their library used by *everyone*, including billionaires. They choose **MIT**.
- **Example B**: A developer wants to ensure that all improvements to their software remain free and open. They choose **GPLv3**.
- **Example C**: A company wants to share their code but protects their patents. They choose **Apache 2.0**.

## 💬 Good vs. Bad Communication
**Scenario**: You found a bug in a React component.

**❌ Bad Comment:**
> "It's broken. Fix it soon."

**✅ Good Comment:**
> "Hi! I noticed that the `Button` component doesn't render correctly on iOS Safari (v16.1). I've attached a screenshot and the error log. I'd love to help fix this if pointed in the right direction!"
