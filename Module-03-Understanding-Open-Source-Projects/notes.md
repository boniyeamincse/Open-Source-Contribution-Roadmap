# Module 3 Notes: Understanding Open Source Projects

## 🗺️ Navigating the Repository
Every professional open-source project follows a similar structure. Understanding these files will save you hours of confusion.

### 📚 Essential Files:
1.  **`README.md`**: The landing page. It explains what the project does, how to install it, and how to use it.
2.  **`CONTRIBUTING.md`**: This is your Bible as a contributor. It contains the rules for submitting code, setup instructions, and the code style guide.
3.  **`LICENSE`**: Defines how the software can be used. Common ones include **MIT** (very permissive), **Apache 2.0**, and **GPL** (more restrictive).
4.  **`CODE_OF_CONDUCT.md`**: Standards for community behavior to ensure a healthy, welcoming environment.

---

## 🚦 Finding Work: The Issue Tracker
Projects use "Issues" to track bugs and tasks.
-   **Labels**: Look for labels like `good first issue`, `beginner-friendly`, or `help wanted`.
-   **Conversation**: Read the comments! If someone is already working on it, find another one.
-   **Claiming an Issue**: Once you find an issue you can solve, comment: *"I'd like to work on this. Could you assign it to me?"*

---

## ⚖️ Open Source Licenses (Simplified)
-   **Permissive (MIT/Apache)**: Use it for almost anything, just give credit.
-   **Copyleft (GPL)**: If you use this code in your app, your app must also be open-source.
-   **Public Domain (Unlicense)**: No strings attached.

---

## 🗣️ Community Etiquette
-   **Be Polite**: Remember maintainers are often volunteers. Use "please" and "thank you".
-   **Provide Context**: When reporting a bug, explain your OS, version, and steps to reproduce.
-   **Respect Time**: Don't tag maintainers unnecessarily. Give them a few days to respond.

---

## 💡 Summary
A repository is a community space. Reading the documentation and following the rules shows that you respect the time and effort of others.
